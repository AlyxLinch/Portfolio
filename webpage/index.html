The brief

A visual system, not a decorative loop.

The first plan was practical: render a beautiful looping video and use it as a lightweight background. The more interesting answer was to make the scene respond to the page itself.

The final experience keeps the waveform alive while scroll moves the camera through a spiral path in 3D space. A separate tuner exposes the wave, camera, palette, path, texture, and post-processing parameters, so visual decisions can be tested in context instead of guessed in code.

The evolution

The useful version came from experiments, not a straight line.

These frames reconstruct the major visual stages from the project’s settings and development history. Some are recreations, because early prototypes were treated like sketches: useful, messy, and not always archived before the next idea arrived.

01

Starting point

Two mathematical sources

A Wolfram Demonstration gave me the seed: two cosine waves radiating from separate points and combining into one interference surface. I translated the equation into a Three.js mesh and stripped out random noise so the underlying motion stayed crisp and readable.

  • Radial interference
  • Two sources
  • Vertex displacement
02

Finding a voice

From symmetry to a folded landscape

Perfect rings were satisfying, but they still read as a simulation. I wanted something with more tension, so I folded and sheared the coordinates, then blended in directional movement. The surface started to feel less like a physics demo and more like a living brand element.

  • Asymmetric fold
  • Camera framing
  • Height-mapped color
03

Texture studies

Grain exposed a color problem

I tested static screen noise, multiplied luminance, surface-mapped grain, and palette-aware color texture. The lesson was quick and humbling: black grain made the scene muddy, and oversized speckles looked tiled. The strongest direction nudged fragments along the existing color ramp instead of simply darkening the image.

  • Surface mapping
  • Palette perturbation
  • Luminance preservation
04

Final direction

The camera became the interaction

The breakthrough was letting go of the export. Instead of baking the motion into video, scroll maps to a spiral camera function while the look-at point stays fixed. Raw scroll input is eased before it reaches the camera, so the movement feels intentional even when the user’s trackpad does not.

  • Scroll smoothing
  • Spiral camera path
  • Adaptive quality

The tuning interface

I needed a visual instrument, not a file full of mystery numbers.

The tuner became the project’s control room. It exposes the camera, scroll path, mesh density, wave behavior, palette, grain, and post-processing settings. Guide frames show how the composition lands across desktop, tablet, and mobile before a value is saved back to the shared config.

What did not ship

Dead ends made the final product smaller and clearer.

A

A 90-second video fallback

I built frame-stepped capture tools, ffmpeg streaming, WebCodecs experiments, diagnostic logs, headless Chromium rendering, and capture completion states. The pipeline worked, but the file size fought the concept. Worse, video removed the best part: moving through the scene.

Decision: remove the fallback entirely.
B

Device capability before interaction

One prototype locked scroll while a hidden renderer benchmarked the device, then chose between video and WebGL. It solved the camera-snap problem on paper, but it made the first few seconds feel overmanaged.

Decision: start live and adapt continuously.
C

Grain as the defining visual effect

Grain looked beautiful in references, especially the soft retro airbrushed examples. On the live surface, it also introduced aliasing, banding, muddy luminance, and scale issues. I kept texture as a tuning option, but stopped asking it to carry the whole identity.

Decision: treat texture as optional, not structural.

Working with Codex

I used AI as an implementation partner, not an idea generator.

1Observe

I described what felt off in visual language: muddy, too fast, snapping, tiled.

2Instrument

Codex turned those observations into controls, logs, counters, and diagnostics.

3Compare

I tested changes live, then came back with specific feedback and visual references.

4Converge

Useful experiments became shared settings. Brittle branches got retired.

“The most useful prompts were not vague requests for polish. They were specific descriptions of perception: the camera feels too high, the grain is mapped too large, the frame darkens when the effect turns on.”

Wave model

Displacement before decoration

Each mesh vertex samples the selected wave function. Height values are normalized every frame, so the full palette remains visible even when amplitude, damping, or camera angle changes.

Camera model

One path, two interfaces

The tuner’s path scrubber and the live page call the same spiral-camera function. That removed the framing mismatch that appeared when each interface calculated camera movement on its own.

Performance model

Quality can move both directions

The renderer targets 30 fps. It reduces mesh density before device pixel ratio, never drops below 300 segments, and raises quality again after the device shows sustained headroom.

Post-processing

Effects without a darker frame

Chromatic aberration, diffusion, and halation use a multisampled full-resolution render target. Matching Three.js color-space conversion kept the post-process pass from quietly dimming the entire image.

Outcome

The final background is also the documentation.

As you scroll this page, the background follows the same camera path described above. The performance readout is intentionally visible in this prototype, so the design claim stays inspectable instead of becoming a hand-wavy flourish.
